## 二、致命缺陷（Blockers）

### 🚨 Blocker 1：Skill 自身犯了"教条主义"——用历史隐喻替代技术判断

**问题描述**：
本 Skill 批评"教条主义"是"照搬大厂微服务架构，不考虑团队规模"，但自身犯了更严重的教条主义——**用历史叙事替代技术决策框架**。

**具体表现**：
- 9 个阶段完全按长征时间线排列，而非按工程迁移的实际逻辑顺序排列
- "四渡赤水"被映射为"敏捷迭代、灰度发布"，但四渡赤水的本质是**战略欺骗和机动迂回**，与工程中的"灰度发布"逻辑完全不同
- "翻雪山过草地"被映射为"极限环境测试"，但长征中的雪山草地是**被动承受的恶劣环境**，而非主动设计的测试环境

**风险**：
- 团队成员可能为了"贴合长征隐喻"而做出错误的技术决策
- 历史事件的"悲壮感"可能掩盖技术决策的理性分析
- 隐喻的强制性可能导致"为了用隐喻而用隐喻"的技术自嗨

**建议**：
- 删除或弱化历史叙事，将 9 阶段重构为**纯技术决策流程**
- 保留"长征精神"作为文化层，但技术层必须独立
- 增加"何时不应使用本 Skill"的明确判断标准

---

### 🚨 Blocker 2：Skill 自身犯了"冒险主义"——无灰度直接全量上线

**问题描述**：
本 Skill 在"会师与交付"阶段建议"庆祝阶段性胜利"，但**完全没有定义"阶段性胜利"的验收标准**。

**具体表现**：
- "明确会师标准：定义完成的验收条件"——但模板中只写了"[性能指标]、[覆盖率]、[回滚方案]"的占位符，没有给出任何具体指标
- 没有定义"最小可交付单元"（MVP），可能导致一次性全量上线
- 没有定义"回滚触发条件"，即什么情况下必须放弃当前路线

**对比**：
- refactor-architecture 每个 phase 都有明确的"skip 条件"和"validation gate"
- 本 Skill 只有"[可量化指标]"的占位符，没有量化标准

**建议**：
- 为每个 phase 定义具体的**通过/失败标准**
- 增加"最小可交付单元"定义
- 增加"回滚触发条件"（如：核心交易数据丢失 > 0.1% 立即回滚）

---

### 🚨 Blocker 3：Skill 自身犯了"分裂主义"——与已有 Skill 冲突

**问题描述**：
用户已有多个 Skill（PHP 代码审查、Kafka DLQ、支付对账等），本 Skill 的"四种反模式"命名可能与已有 Skill 的规范冲突。

**具体表现**：
- 本 Skill 定义"教条主义 = 照搬大厂微服务架构"，但用户的 PHP 代码审查 Skill 可能定义了不同的"教条主义"
- "逃跑主义 = 遇到性能问题就喊重写"——但用户的 Kafka 支付多样性研究中，可能确实需要重写某些模块
- 多个 Skill 对同一概念的不同定义，会导致团队认知混乱

**风险**：
- 团队在不同 Skill 中看到对"教条主义"的不同定义，产生困惑
- 架构评审时，不同 Skill 的引用可能导致决策冲突

**建议**：
- 在 Skill 中增加"与已有 Skill 的兼容性声明"
- 将反模式命名改为**技术术语**（如"架构教条主义"→"Context-Free Architecture Adoption"），避免与政治隐喻混淆
- 增加"本 Skill 的反模式定义仅适用于迁移场景"的限定

## 三、严重缺陷（Critical）

### ⚠️ Critical 1：完全不适用于 PHP/Hyperf 技术栈

**问题描述**：
本 Skill 的模板和示例完全基于通用概念，没有考虑用户的技术栈（PHP 8.x / Hyperf 3.2 / Swoole / Kafka / MySQL）。

**具体表现**：
- "暗线布局：数据双写"——在 Hyperf 中如何实现？是通过 AOP 拦截？还是通过中间件？没有指导
- "影子流量验证"——在 Swoole 协程环境下，如何复制请求而不阻塞主流程？没有提及
- "接口兼容层"——Hyperf 的接口兼容性如何保证？是通过 Protocol Buffers？还是通过 JSON Schema？没有定义
- "数据双写"——MySQL 主从同步延迟如何处理？没有提及
- "分布式事务"——Hyperf 中如何使用 Saga 模式或 TCC 模式？没有提及

**对比**：
- refactor-architecture 明确限定"TS/JS 生态"，并给出具体的 ESM re-export 模式
- 本 Skill 声称适用于"大规模系统重构、技术栈迁移"，但没有限定技术栈，也没有给出任何技术实现细节

**建议**：
- 明确限定适用范围，或增加"技术栈适配指南"
- 为 PHP/Hyperf 生态增加具体实现示例：
  - 数据双写：Hyperf AOP + Kafka 生产者
  - 影子流量：Swoole 协程池 + 异步 HTTP Client
  - 接口兼容：JSON Schema 验证 + 版本路由
  - 分布式事务：Saga 模式实现（Hyperf 的 Transaction 组件）

---

### ⚠️ Critical 2："遵义会议"模板无法在实际会议中使用

**问题描述**：
"遵义会议"模板设计为架构评审会议使用，但模板内容过于抽象，无法直接用于实际会议。

**具体表现**：
- "当前战况（事实数据）"要求填写"系统故障率：[数据]"——但没有定义如何收集这些数据
- "左倾错误识别"要求勾选"是否存在教条主义"——但没有定义判断标准
- "路线修正决策"要求填写"原路线：[描述]"——但没有定义路线描述的格式
- 没有会议议程、时间分配、参会人员角色定义

**对比**：
- c3-skill 的"eval 绑定"将架构文档绑定到具体的代码文件和检查命令
- 本 Skill 的模板只有占位符，没有操作指南

**建议**：
- 为"遵义会议"模板增加：
  - 会议前准备清单（数据收集脚本、监控面板截图、故障报告模板）
  - 会议议程（30 分钟数据回顾 + 30 分钟错误识别 + 30 分钟决策 + 15 分钟行动计划）
  - 参会人员角色（决策者/记录者/反对者/观察者）
  - 决策记录格式（ADR - Architecture Decision Record）

---

### ⚠️ Critical 3："泸定桥突击队"模板缺乏资源约束定义

**问题描述**：
"组建突击队：挑选最精锐的 2-3 人攻坚核心模块"——但没有定义"精锐"的标准，也没有定义资源约束。

**具体表现**：
- 2-3 人是否包含测试人员？是否包含运维人员？
- "限时攻坚"的 deadline 如何设定？是基于业务 deadline 还是技术评估？
- 如果突击队成员被其他紧急任务抽调，如何处理？
- 攻坚期间，突击队的日常维护工作由谁承担？

**建议**：
- 增加"突击队组建规范"：
  - 人员构成：1 架构师 + 1 核心开发 + 1 测试（可选）
  - 资源隔离：攻坚期间不参与其他项目，日常维护由"牵制部队"承担
  - Deadline 设定：技术评估时间 × 1.5 倍缓冲
  - 人员替补：指定 1 名替补成员

## 四、中等缺陷（Major）

### 🟡 Major 1：没有定义"转移目标"的选择决策树

**问题描述**：
Phase 1 要求"明确转移目标：是重构？迁移？还是重写？"，但没有给出选择标准。

**建议**：
增加决策树：
```
旧系统是否还能运行？
├─ 否 → 重写（紧急止血）
└─ 是 → 技术债务是否可局部修复？
    ├─ 是 → 重构（局部优化）
    └─ 否 → 迁移（渐进式替换）
```

---

### 🟡 Major 2：没有考虑"多轮迁移"场景

**问题描述**：
长征是一次性的战略转移，但工程迁移往往是多轮次的（如：先迁移支付模块，再迁移订单模块）。

**建议**：
- 增加"多轮迁移"的循环机制
- 每轮迁移结束后，是否回到 Phase 1 重新评估？
- 如何防止"疲劳迁移"（团队在连续多轮迁移后 burnout）？

---

### 🟡 Major 3："根据地建设"阶段过于薄弱

**问题描述**：
"根据地建设"只有 3 个检查项（监控、告警、文档），而实际运维需要：
- 容量规划（Swoole Worker 数量、MySQL 连接池大小）
- 性能基线（P50/P95/P99 响应时间）
- 故障演练（Chaos Engineering）
- 数据归档策略

**建议**：
- 将"根据地建设"扩展为完整的"运维治理框架"
- 增加 Hyperf 特有的监控指标（Swoole 协程数、连接池状态、Kafka 消费延迟）

---

### 🟡 Major 4：没有考虑"合规与审计"维度

**问题描述**：
用户是 RastarGame，涉及支付对账、SDK 订单状态修改，必须考虑：
- 数据迁移的审计日志（谁、何时、修改了什么）
- 支付数据的合规性（PCI DSS、GDPR）
- 回滚操作的可追溯性

**建议**：
- 在每个 phase 中增加"合规检查项"
- 在"暗线布局"阶段增加"审计日志双写"
- 在"根据地建设"阶段增加"合规审计报告"

## 六、反模式自检：本 Skill 犯了哪些"左倾错误"？

### 1. 教条主义 ✅ 犯

**证据**：
- 用长征的 9 个历史阶段强制映射到工程流程，不考虑项目实际复杂度
- 要求所有迁移项目都必须经历"翻雪山过草地"（极限环境测试），但某些内部系统迁移可能不需要
- "四渡赤水"（灵活机动）被强制映射为"灰度发布"，但某些项目可能更适合蓝绿部署

---

### 2. 冒险主义 ✅ 犯

**证据**：
- "飞夺泸定桥"（限时攻坚）鼓励在 deadline 压力下强行推进，但没有定义"什么情况下必须放弃攻坚"
- "接受湘江损失"（接受数据损失）没有定义"可接受损失的上限"
- 没有定义"回滚触发条件"，可能导致"明知会失败还继续推进"

---

### 3. 逃跑主义 ✅ 犯

**证据**：
- "遇到无法快速解决的旧系统 Bug，先绕过"——这本身就是"逃跑主义"
- 但 Skill 将"逃跑主义"定义为"遇到性能问题就喊重写"，存在自我矛盾
- 没有定义"绕过"的临时方案必须包含"后续修复计划"

---

### 4. 分裂主义 ✅ 犯

**证据**：
- 本 Skill 的"四种反模式"命名与常见技术术语冲突，可能导致团队认知分裂
- "教条主义"在技术语境中通常指"盲目遵循设计模式"，而本 Skill 将其定义为"照搬大厂微服务架构"
- 没有与其他 Skill（如 PHP 代码审查）的反模式定义对齐

---

## 七、改进建议（优先级排序）

### P0（必须修复）

1. **重构 9 阶段为技术决策流程**：删除历史隐喻的强制性，保留为可选文化层
2. **为每个 phase 定义通过/失败标准**：给出具体的量化指标和 skip 条件
3. **增加技术栈适配指南**：为 PHP/Hyperf/Swoole/Kafka/MySQL 提供具体实现示例
4. **修复反模式自我矛盾**：重新定义"逃跑主义"，增加"绕过必须附带修复计划"

### P1（强烈建议）

5. **重写"遵义会议"模板**：增加会议议程、角色定义、ADR 格式
6. **重写"泸定桥突击队"模板**：增加人员构成、资源隔离、替补机制
7. **增加"转移目标"决策树**：重构/迁移/重写的选择标准
8. **增加合规与审计维度**：支付数据的审计日志、合规检查项

### P2（建议）

9. 增加短触发词 `/lmr`
10. 拆分 history.md 为按需加载的多个文件
11. 增加版本升级路径和 evolution-log 机制
12. 增加"多轮迁移"的循环机制
