# 示例 4：失败案例 — 跳过 Phase 2 导致数据丢失

> 场景：某游戏公司（非 RastarGame）决定从 ThinkPHP 5.0 迁移到 Hyperf 3.2，团队 3 人，项目周期 2 个月。
> 
> **关键错误**：项目经理为了"赶进度"，跳过 Phase 2（突围准备与暗线部署），直接从 Phase 3（初期迁移）开始全量切换。
> 
> 本案例展示了一个真实的失败路径，以及如何避免。

---

## 背景

- **旧系统**：ThinkPHP 5.0 + MySQL 5.6 + Redis 3.0，运行 3 年，约 5 万行 PHP 代码
- **新系统**：Hyperf 3.2 + Swoole 5.x + MySQL 8.0 + Redis 6.0
- **团队**：3 人（1 架构师 + 2 开发）
- **业务**：游戏内购系统（支付、订单、道具发放）
- **项目周期**：2 个月（2026-03-01 至 2026-04-30）

---

## Phase 1: 危机评估与转移决策（执行正确）

### 问题根因报告

| 问题 | 数据 | 根因 |
|------|------|------|
| ThinkPHP 5.0 已 EOL | 官方停止维护，最后更新 2023 年 | 技术债务 |
| Swoole 4.x 不兼容 PHP 8.2 | 编译失败 | 依赖锁死 |
| 单文件 3000 行 | `PaymentController.php` 3000 行，包含支付、退款、对账、统计 | 代码耦合 |
| 性能瓶颈 | 支付回调 P95 延迟 800ms，高峰期超时 | 架构瓶颈 |

### 迁移策略选择

根据决策树：
- 旧系统还能运行？→ 是（但性能差）
- 技术债务是否可局部修复？→ 否（框架 EOL 无法局部修复）
- **结论：渐进式迁移（Strangler Fig）**

**正确决策**：选择渐进式迁移，计划分 3 轮（支付 → 订单 → 道具）。

---

## Phase 2: 突围准备与暗线部署（被跳过！）

### 原计划

- 部署数据双写：ThinkPHP 写入 MySQL 后，通过 AOP 拦截，异步发送到 Kafka
- 部署接口兼容层：Nginx 根据 Header 路由到 ThinkPHP 或 Hyperf
- 部署影子流量：复制请求到 Hyperf，对比响应结果
- 与支付渠道（微信、支付宝、苹果）确认接口兼容性

### 实际执行（错误！）

**项目经理决策**："时间紧，Phase 2 太复杂，直接做 Phase 3。数据双写后面补，先用全量切换，出问题再回滚。"

**跳过 Phase 2 的理由**：
1. "数据双写太复杂，Kafka 我们还没用过"
2. "接口兼容层没必要，前端直接改 URL 就行"
3. "影子流量浪费时间，直接上线看效果"
4. "支付渠道接口不会变，不用确认"

---

## Phase 3: 初期迁移与损失控制（灾难发生）

### 全量切换（2026-03-15）

- 没有灰度，没有数据双写，没有接口兼容层
- 前端直接修改 URL，从 ThinkPHP 切换到 Hyperf
- 支付渠道回调地址从 `https://api.old.com/pay/callback` 改为 `https://api.new.com/pay/callback`

### 灾难时间线

| 时间 | 事件 | 影响 |
|------|------|------|
| 2026-03-15 10:00 | 全量切换上线 | 支付成功率 99.8%（正常） |
| 2026-03-15 14:00 | 微信支付回调延迟 | 部分用户支付后未收到道具 |
| 2026-03-15 16:00 | 支付宝回调格式不兼容 | Hyperf 解析失败，订单状态未更新 |
| 2026-03-15 18:00 | 用户投诉激增 | 客服收到 200+ 投诉，"支付成功但没收到道具" |
| 2026-03-15 20:00 | 紧急回滚到 ThinkPHP | 回滚耗时 2 小时（没有回滚脚本，手动改 Nginx 配置） |
| 2026-03-15 22:00 | 数据不一致发现 | 回滚后，ThinkPHP 数据库和 Hyperf 数据库数据不一致 |
| 2026-03-16 02:00 | 数据修复开始 | 团队通宵对比数据库，修复不一致数据 |
| 2026-03-16 10:00 | 数据修复完成 | 修复 1200 条订单数据，3 条数据无法修复（Hyperf 写入后 ThinkPHP 未同步） |
| 2026-03-16 12:00 | 业务方问责 | 项目经理被要求写事故报告 |

### 损失统计

| 损失类型 | 数量 | 说明 |
|---------|------|------|
| 用户投诉 | 200+ 起 | 支付成功但未收到道具 |
| 数据不一致 | 1200 条 | 订单状态、支付状态不一致 |
| 无法修复数据 | 3 条 | Hyperf 写入后 ThinkPHP 未同步，数据丢失 |
| 回滚时间 | 2 小时 | 没有回滚脚本，手动操作 |
| 团队通宵 | 1 晚 | 3 人通宵修复数据 |
| 项目延期 | 2 周 | 原计划 2 个月，实际 2.5 个月 |
| 团队士气 | 严重受损 | 2 名开发对迁移项目产生恐惧，1 人提出离职 |
| 业务方信任 | 严重受损 | 业务方要求"以后任何技术变更必须提前 2 周通知" |

## 复盘：为什么会失败？

### 直接原因

1. **没有数据双写**：Hyperf 和 ThinkPHP 使用不同的数据库，数据没有同步。Hyperf 写入的订单，ThinkPHP 看不到。
2. **没有接口兼容层**：前端直接改 URL，没有 fallback 机制。Hyperf 出问题时，无法快速切回 ThinkPHP。
3. **没有与支付渠道确认**：支付宝回调格式在 Hyperf 中解析失败（ThinkPHP 做了隐式格式转换，Hyperf 没有）。
4. **没有回滚脚本**：回滚需要手动改 Nginx 配置，耗时 2 小时。
5. **没有灰度**：全量切换，问题影响所有用户。

### 根本原因

1. **项目经理的"进度优先"思维**：为了赶 2 个月 deadline，跳过"准备工作"，认为"出问题再回滚就行"。
2. **对迁移复杂度的低估**：认为"只是换个框架，代码逻辑一样"，忽略了数据同步、接口兼容、第三方依赖等隐性成本。
3. **团队经验不足**：3 人团队没有 Kafka 使用经验，也没有迁移经验，对 Phase 2 的工作量缺乏认知。
4. **业务方压力**：业务方要求"2 个月内必须完成"，导致团队被迫压缩必要的准备时间。

## 如果按本 Skill 执行，应该如何做？

### Phase 2（被跳过的阶段）正确执行方案

**数据双写**：
- ThinkPHP 写入 MySQL 后，通过 AOP 拦截（ThinkPHP 的 Behavior 机制），异步发送到 Kafka
- Hyperf 消费 Kafka 消息，写入新数据库
- 旧系统和新系统使用**不同的数据库**（避免互相影响）
- 双写运行 2 周，验证数据一致性 99.99%

**接口兼容层**：
- Nginx 层路由：根据 Header `X-API-Version` 路由到 ThinkPHP 或 Hyperf
- 默认路由到 ThinkPHP（v1），灰度阶段逐步切换到 Hyperf（v2）
- 响应格式保持不变（JSON 结构、字段名、错误码）
- 前端无需修改 URL，只需在 Header 中指定版本

**影子流量**：
- ThinkPHP 处理请求后，复制请求参数和响应结果到 Kafka `shadow-traffic-topic`
- Hyperf 消费影子流量，模拟处理，对比响应结果
- 仅对比响应结构（字段名、类型），不对比动态字段（时间戳、随机 ID）
- 影子流量运行 1 周，响应差异率 < 0.1%

**与支付渠道确认**：
- 提前 2 周与微信支付、支付宝、苹果支付确认回调接口格式
- 在测试环境验证 Hyperf 的回调解析逻辑
- 准备兼容性处理（如：支付宝回调的隐式格式转换）

### Phase 3（初期迁移）正确执行方案

**灰度计划**：
- 第 1 周：10% 流量（内部员工 + 1% 真实用户）
- 第 2 周：30% 流量（新增 10% 真实用户）
- 第 3 周：50% 流量（新增 20% 真实用户）
- 第 4 周：100% 流量（全部切换）

**可控损失**：
- 非核心报表延迟：< 5 分钟（提供缓存数据快照）
- 核心交易数据丢失：**0%**（数据双写确保）

---

## 关键教训

1. **Phase 2 不可跳过**：数据双写、接口兼容层、影子流量是迁移的"安全带"，没有安全带的迁移等于"裸奔"。
2. **"赶进度"是最大陷阱**：跳过准备阶段省下的 2 周，会导致后期修复损失 2 周 + 团队士气受损 + 业务方信任丧失。
3. **全量切换 = 赌博**：没有灰度的全量切换，是在用所有用户做赌注。
4. **回滚脚本必须在上线前准备好**：回滚不是"出问题再想办法"，而是上线前必须演练的操作。
5. **第三方依赖必须提前确认**：支付渠道、客服平台、第三方 SDK 的接口兼容性，必须在 Phase 2 确认。

---

## 本 Skill 的防御机制

如果该团队使用了本 Skill，以下机制会阻止失败：

1. **适用性自检**：第 4 题"业务方是否知情并同意？"——如果业务方要求"2 个月必须完成"，团队应拒绝或调整范围，而非跳过 Phase 2。
2. **Phase 2 通过标准**："数据双写机制已部署"——如果不满足，无法进入 Phase 3。
3. **Phase 3 失败标准**："新系统故障率 > 旧系统历史故障率 P95 × 2"——如果故障率超标，触发回滚到 Phase 2。
4. **Rollback 方案**："流量切换回旧系统（接口兼容层秒级切换）"——如果有兼容层，回滚只需 30 秒，而非 2 小时。
5. **反模式自检**："Big-Bang Deployment（大爆炸部署）"——全量切换是典型的反模式，Skill 会明确警告。
