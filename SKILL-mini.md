# Skill: 长征工程方法论 (Long March Engineering Methodology) — 极简版

> **版本**: 1.3.1-mini
> **说明**: 极简版单文件，仅包含核心哲学、适用性自检、Phase 1-9 极简版（仅通过/失败标准）、反模式。适合快速查阅和紧急决策。
> **完整版**: 如需详细模板、示例、监控指南，请使用 SKILL-lite.md 或分文件版

---

## 核心哲学

1. **生存优先**：系统能跑 > 架构优雅。先止血，再治病。
2. **路线灵活性**：没有唯一正确的架构，只有适应当下约束的架构。
3. **团队可持续性**：迁移是马拉松，不是冲刺。每完成一个里程碑，必须给团队技术债务清理 Sprint。

---

## 适用性自检

以下 4 个问题必须全部回答"是"：

1. [ ] 旧系统是否已出现**无法局部修复**的系统性问题？
2. [ ] 团队是否已达成**"必须迁移"的共识**？
3. [ ] 根据迁移规模，人力资源是否充足？
   - 微型（1 人 1 周）：1 人即可
   - 小型（2 人 2-4 周）：至少 2 人
   - 中型（3-5 人 1-3 个月）：至少 3 人
   - 大型（5+ 人 6-12 个月）：至少 5 人
4. [ ] 业务方是否**知情并同意**迁移可能带来的阶段性影响？

如果第 1-2 题或第 4 题回答"否" → **不要使用本 Skill**。
如果第 3 题回答"否" → **缩小迁移规模**或**暂缓迁移**。

## 九阶段极简版

### Phase 1: 危机评估与转移决策

**目标**：确认迁移必要性，选择迁移策略（重构/迁移/重写）。

**通过标准**：
- [ ] 旧系统问题根因报告已产出（含监控数据、故障记录）
- [ ] 迁移策略已选定（重构/迁移/重写）
- [ ] 迁移范围已明确（哪些模块迁移、哪些保留、哪些废弃）
- [ ] 业务方已知情同意

**失败标准（回滚）**：
- 旧系统问题可通过局部优化解决
- 团队未达成共识
- 业务方不接受任何影响

**Skip**：旧系统已完全不可用 → 直接 Phase 3（紧急重写）

**决策树**：
- 旧系统不能运行 → 重写（72 小时恢复核心流程）
- 旧系统能运行 + 技术债务可局部修复 → 局部优化（1-2 人，2 周）
- 旧系统能运行 + 技术债务不可局部修复 → 渐进式迁移（3-5 人，6-12 个月）

---

### Phase 2: 突围准备与暗线部署

**目标**：完成技术预研、数据双写、接口兼容层、影子流量验证。

**通过标准**：
- [ ] 新系统技术栈 PoC 已通过
- [ ] 数据双写机制已部署，旧系统写入成功率 100%
- [ ] 接口兼容层已部署，旧系统调用方可无缝切换
- [ ] 影子流量已运行 7 天，差异率 < 阈值

**失败标准（回滚）**：
- PoC 无法运行核心业务流程
- 数据双写导致旧系统性能下降 > 阈值

**Skip**：采用"重写"策略 → 跳过数据双写和影子流量

**牵制部队**（为 Phase 6 做准备）：
- 人员：1-2 人，熟悉旧系统
- 职责：攻坚期间承担日常维护、Bug 修复、报警响应
- 交接：Phase 2 结束前，核心团队向牵制部队交接旧系统维护文档

---

### Phase 3: 初期迁移与损失控制

**目标**：首批模块迁移，接受可控损失，保留核心骨干。

**通过标准**：
- [ ] 首批模块（非核心但高频）已上线
- [ ] 旧系统流量已切换 10% 到新系统，无异常
- [ ] 核心交易数据 100% 强一致，无丢失
- [ ] 非核心数据允许延迟/丢失 < 阈值

**失败标准（回滚）**：
- 核心交易数据丢失 > 0
- 新系统故障率 > 旧系统 × 2
- 团队核心成员 burnout

**Skip**：采用"重写"策略 → 无灰度，直接全量上线核心链路

---

### Phase 4: 里程碑复盘与路线修正

**目标**：数据驱动的架构评审，修正错误决策。

**通过标准**：
- [ ] 里程碑数据已收集（故障率、性能、团队反馈、业务影响）
- [ ] 架构评审会议已召开，时长 2 小时，参会 ≥ 5 人
- [ ] 会议产出《路线修正决议》（ADR），含原路线、修正后路线、决策人、执行时限
- [ ] 团队对修正后路线达成共识

**失败标准（回滚）**：
- 会议无数据支撑，纯主观讨论
- 会议无决策人拍板，陷入无限期争论

**Skip**：前 3 个 Phase 数据指标全部达标 → 跳过复盘，直接进入 Phase 5

**决策人资格**（通用标准，适用于 Phase 4 和 Phase 7）：
- 必须理解当前技术栈
- 必须阅读过双方方案文档
- 必须要求双方提供数据验证
- 必须给予双方同等时间陈述
- 必须对决策结果负责

---

### Phase 5: 敏捷迭代与方案切换

**目标**：通过灰度发布、AB 测试、快速切换验证假设。

**通过标准**：
- [ ] 灰度发布已部署，流量切换 10% → 30% → 50% → 100%，每阶段稳定 3 天
- [ ] AB 测试已运行，核心指标差异 < 阈值
- [ ] 方案切换机制已验证（5 分钟内可回滚）

**失败标准（回滚）**：
- 灰度阶段新系统错误率 > 旧系统 × 1.5
- 客户投诉量上升 > 阈值
- 切换机制无法在 5 分钟内完成

**Skip**：模块无用户可见影响 → 跳过 AB 测试，仅做灰度

---

### Phase 6: 核心瓶颈限时攻坚

**目标**：集中最强资源，在明确 deadline 内解决核心性能瓶颈或关键 Bug。

**通过标准**：
- [ ] 核心问题已明确（如：支付回调延迟 > 阈值）
- [ ] 攻坚小组已组建（1 架构师 + 1 核心开发 + 1 测试，可选）
- [ ] Deadline 已设定（技术评估 × 1.5，最长 2 周）
- [ ] 攻坚期间，小组成员不参与其他项目，日常维护由**牵制部队**承担
- [ ] 问题已解决，验证通过（性能提升 > 50% 或 Bug 修复率 100%）

**失败标准（回滚）**：
- Deadline 到期未解决，且未提前 3 天预警
- 攻坚期间引入新 Bug，导致其他模块故障
- 攻坚小组成员 burnout

**Skip**：前 5 个 Phase 无核心瓶颈 → 跳过此阶段

---

### Phase 7: 技术分歧快速裁决

**目标**：技术争论超过 2 周无结论时，由决策人拍板。

**通过标准**：
- [ ] 技术分歧已明确（双方方案已书面化，含优缺点分析）
- [ ] 争论时长已确认（超过 2 周或 3 次会议无结论）
- [ ] 决策人已听取双方意见，并做出最终决定
- [ ] 反对意见已记录（即使被否决）
- [ ] 核心团队未分裂（所有成员继续执行决策）

**失败标准（回滚）**：
- 决策人未在 48 小时内做出决定
- 决策后仍有团队成员拒绝执行，导致项目停滞
- 决策导致核心数据或核心产品受损

**Skip**：分歧在 2 周内通过数据验证解决 → 无需裁决

---

### Phase 8: 多团队协作验收

**目标**：定义验收标准，完成交付，给团队技术债务清理 Sprint。

**通过标准**：
- [ ] 验收标准已定义（功能完整性、性能指标、数据一致性、监控覆盖、告警有效、文档完整、回滚可行、合规审计）
- [ ] 所有模块已通过验收（或明确标记为"遗留问题"并分配负责人）
- [ ] 多团队已签署《交付确认书》（各团队负责人确认）
- [ ] 团队技术债务清理 Sprint 已安排（建议 3-5 天或 1 个 Sprint 的 20%）

**失败标准（回滚）**：
- 核心模块未通过验收，但被迫交付
- 无技术债务清理 Sprint，团队立即投入下一项目
- 遗留问题未分配负责人，成为"技术债务黑洞"

**Skip**：单人项目 → 无需"多团队协作验收"，但必须有"自我验收清单"

---

### Phase 9: 运维治理与资产沉淀

**目标**：从"项目模式"转向"产品模式"，建立监控、告警、文档、合规审计体系。

**通过标准**：
- [ ] 监控体系已建立（关键路径 100% 覆盖）
- [ ] 告警规则已配置（准确率 > 90%，噪音率 < 50%）
- [ ] 运维文档已完整（架构、API、故障处理、回滚手册）
- [ ] 技术资产已沉淀（复用组件、工具脚本、配置模板）
- [ ] 合规审计已通过（核心操作 100% 有审计日志）
- [ ] 容量规划已制定（未来 6 个月增长预测 + 扩容计划）
- [ ] 故障演练已执行（Chaos Engineering）

**失败标准（回滚）**：
- 监控缺失导致故障发现延迟 > 阈值
- 告警噪音率 > 50%（团队麻木，忽略告警）
- 合规审计未通过（核心操作缺失审计日志）

**Skip**：内部工具无合规要求 → 跳过"合规审计"；流量 < 100 QPS → 跳过"容量规划"和"故障演练"

## 反模式

### 1. Context-Free Architecture Adoption（无上下文架构采纳）
照搬大厂架构，不考虑团队规模、维护能力和业务复杂度。
**修复**：单体优先，按需拆分。

### 2. Big-Bang Deployment（大爆炸部署）
不做灰度，直接全量上线，赌"不会出问题"。
**修复**：强制灰度 10%→30%→50%→100%，回滚演练。

### 3. Rewrite-First Syndrome（重写优先综合征）
遇到问题不分析根因，直接喊"重写"。
**修复**：强制 RCA，渐进式重写，知识迁移。

### 4. Analysis Paralysis（分析瘫痪）
技术争论超过 2 周或 3 次会议无结论。
**修复**：2 周后强制决策，记录反对意见，数据验证优先。

## 阈值计算前提条件（通用）

| 前提条件 | 要求 | 不满足时的 Fallback |
|---------|------|-------------------|
| 样本量 | >= 1000 个数据点 | 使用行业经验值作为临时阈值，同时增加采样频率 |
| 时间窗口 | >= 2 周 | 使用固定阈值，持续积累数据后替换 |
| 数据稳定性 | 排除异常时段（促销、DDoS、版本发布） | 手动标注异常时段后重新计算 |
| P95 = 0 或不存在 | 历史从未触发过该指标 | 使用行业经验值作为初始阈值，积累 1000+ 样本后替换 |

**临时阈值（行业经验值）**：
- HTTP P95 延迟：< 500ms（游戏内购 < 200ms，后台 < 1000ms）
- HTTP 错误率：< 0.1%（核心交易 < 0.01%，非核心 < 0.5%）
- DB/Redis 连接池使用率：< 80%
- Swoole 协程数：< 10000
- Kafka 消费延迟：< 1000 条
- 团队压力评分：< 3/5
