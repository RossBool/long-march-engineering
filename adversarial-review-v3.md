## 四、新增的严重缺陷（v1.2.0 引入）

### ⚠️ Critical 1（新增）：文件拆分后的"加载机制"缺失

**状态**：❌ **新增问题**

**问题描述**：
v1.2.0 将 SKILL.md 拆分为 13 个文件，但**没有定义 AI 如何知道加载哪个文件**。

**具体场景**：
- 用户说"/lmr" → AI 加载 SKILL.md（3,106 字符），但不知道用户当前处于哪个 phase
- 用户说"我们正在做架构评审" → AI 需要理解"架构评审 = Phase 4"，然后加载 `phase-04-milestone-review.md`
- 用户说"帮我写个攻坚计划" → AI 需要理解"攻坚计划 = Phase 6 模板"，然后加载 `templates/core-bottleneck.md`

**为什么这是 Critical**：
- 如果 AI 无法正确路由到对应文件，用户需要手动指定文件名（如"加载 phase-04"），这严重降低了用户体验。
- 更糟的是：AI 可能加载错误的 phase 文件，给出错误的指导（如：用户在 Phase 3 初期迁移，AI 却加载了 Phase 9 的运维治理内容）。

**对比**：
- refactor-architecture 的"渐进式上下文加载"是通过 skill 内部逻辑自动管理 phase 状态的。
- 本 Skill 虽然拆分了文件，但没有定义 phase 状态管理和文件路由机制。

**建议修复**：
- 在 SKILL.md 中增加"文件加载路由表"：
  ```
  用户意图 → 加载文件
  "评估是否需要迁移" → phase-01-crisis-assessment.md
  "准备迁移" → phase-02-preparation.md
  "开始迁移" → phase-03-initial-migration.md
  "架构评审/复盘" → phase-04-milestone-review.md + templates/milestone-review.md
  "灰度发布/AB测试" → phase-05-agile-iteration.md
  "解决性能瓶颈" → phase-06-core-bottleneck.md + templates/core-bottleneck.md
  "技术争论/方案选择" → phase-07-tech-dispute.md
  "验收/交付" → phase-08-delivery.md + templates/migration-retrospective.md
  "监控/告警/运维" → phase-09-ops-governance.md
  ```
- 或者：增加一个"状态检测"机制，AI 通过询问用户当前进度来推断 phase

---

### ⚠️ Critical 2（新增）："历史 P95"获取方法缺失，假设团队已有完善监控

**状态**：❌ **新增问题**

**问题描述**：
v1.2.0 的所有阈值都基于"历史 P95"，但**没有提供获取历史 P95 的具体方法**。

**具体表现**：
- Phase 1 的"根因分析脚本"是伪代码，没有提供实际的监控工具配置
- Phase 3 的"可控损失清单"要求"基于历史 P95"，但没有说明如何获取
- Phase 9 的"监控清单"列出了监控项，但没有说明如何部署这些监控

**现实场景**：
- RastarGame 作为游戏公司，可能只有基础的日志系统和简单的告警（如：服务器 CPU > 80% 发短信）
- 没有 Prometheus/Grafana 的成熟部署
- 没有 Swoole 协程数、MySQL 连接池使用率等细粒度监控
- 工程师不知道什么是"P95"，更不知道如何获取

**为什么这是 Critical**：
- 如果团队无法获取"历史 P95"，整个动态阈值体系就崩溃了，所有阈值变成"拍脑袋"。
- 这比 v1.1.0 的"固定阈值"更糟——因为固定阈值至少可以直接使用，而动态阈值需要先有监控基础。

**建议修复**：
- 增加"监控体系建设指南"（作为独立文件 `monitoring-setup.md`）：
  - 阶段 1：基础日志 + 简单告警（1 周可完成）
  - 阶段 2：Prometheus + Grafana 基础部署（2-3 周）
  - 阶段 3：业务指标监控（订单量、支付成功率）（1 周）
  - 阶段 4：Swoole/Hyperf 专用监控（协程数、连接池）（1-2 周）
- 提供"无监控时的 fallback 方案"：如果无法获取 P95，使用固定阈值作为临时方案，同时启动监控建设
- 提供具体的 Hyperf Metric 组件配置示例（伪代码 + 注意事项）

- `templates/migration-retrospective.md`：只有 `[填写]` 占位符，没有示例值

## 五、新增的中等缺陷（v1.2.0 引入）

### 🟡 Major 1（新增）：没有定义"Phase 之间的过渡条件"

v1.2.0 的 9 个 phase 文件是独立的，但**没有定义如何从 Phase N 过渡到 Phase N+1**。

**问题**：
- Phase 1 完成后，如何触发 Phase 2？是自动进入，还是需要人工确认？
- 如果 Phase 3 失败（触发 rollback），是回到 Phase 2 还是 Phase 1？
- 如果 Phase 5 发现需要重新做 Phase 2（暗线部署有问题），如何回溯？

**建议**：增加"Phase 状态机"图或说明，定义：
- 正常流程：Phase 1 → Phase 2 → ... → Phase 9
- 失败回溯：Phase 3 失败 → 回到 Phase 2（修正暗线部署）或 Phase 1（重新评估）
- 跳过规则：Phase 4 数据全部达标 → 跳过 Phase 4，进入 Phase 5

---

### 🟡 Major 2（新增）："牵制部队"概念没有定义交接机制

Phase 2 和 Phase 6 提到了"牵制部队"（维护旧系统的团队），但：
- 没有定义"牵制部队"的人员选拔标准
- 没有定义"牵制部队"和"核心攻坚组"之间的知识交接机制
- 没有定义"牵制部队"的绩效评估（他们做的是"维护旧系统"的脏活，如何激励？）

**建议**：增加"牵制部队管理指南"，包括人员选拔、知识交接清单、激励机制。

### 🟡 Major 3（新增）："休整窗口" rebranding 为"技术债务清理 Sprint"但未在模板中体现

v1.2.0 在审查建议中提到了将"休整窗口" rebranding 为"技术债务清理 Sprint"，但：
- Phase 8 的模板中仍然写的是"休整窗口"
- Phase 8 的交付确认书模板中仍然写的是"休整窗口"
- 没有解释"技术债务清理 Sprint"的具体内容和价值

**建议**：统一术语，在 SKILL.md 和模板中全部改为"技术债务清理 Sprint"，并增加解释说明。

---

### 🟡 Major 4（新增）：没有"多轮迁移"的循环机制

v1.1.0 的审查报告中提到了"多轮迁移"问题，但 v1.2.0 仍未解决：
- 如果项目需要分 3 轮迁移（先支付模块，再订单模块，再用户模块），每轮是否都要走 Phase 1-9？
- 如何避免"疲劳迁移"（团队在连续多轮迁移后 burnout）？
- 每轮之间是否需要"冷却期"？

**建议**：在 SKILL.md 中增加"多轮迁移指南"，定义：
- 每轮迁移独立走 Phase 1-9，但 Phase 1 可简化（复用上一轮的经验）
- 轮间冷却期：至少 1 个 Sprint（2 周）的"技术债务清理 Sprint"
- 疲劳检测：第 3 轮开始前，团队压力评分若 < 3/5，暂停迁移，增加人员或延长期限

---

### 🟡 Major 5（新增）：没有定义"决策人"的权限边界

Phase 4 和 Phase 7 提到了"决策人"，但：
- 如果决策人选择的方案被数据证明错误，谁来承担责任？
- 如果决策人拒绝听取反对意见，如何制衡？
- 如果决策人是 CTO，但 CTO 不懂技术细节，他的决策是否有效？

**建议**：增加"决策人问责机制"：
- 决策人必须签署"决策记录"（ADR 中署名）
- 决策错误后，决策人必须在下次复盘时公开分析错误原因（不追责，但要求复盘）
- 如果决策人连续 2 次决策错误，暂停其决策权 1 轮，由技术委员会集体决策

### 7. 新增反模式：Golden Path Assumption（黄金路径假设） ✅ v1.2.0 新犯

**定义**：假设用户会按照"理想路径"使用本 Skill，忽略了现实中的混乱和偏差。

**证据**：
- v1.2.0 假设用户会"按顺序"走 Phase 1 → Phase 2 → ... → Phase 9
- 但现实中，用户可能从 Phase 3 开始（迁移已启动，才发现需要复盘），或从 Phase 6 开始（突然遇到性能瓶颈）
- v1.2.0 没有提供"中途接入"的指南
- v1.2.0 假设用户已有监控系统可以获取"历史 P95"，但现实中很多团队监控薄弱

---

### 8. 新增反模式：Perfect Template Fallacy（完美模板谬误） ✅ v1.2.0 新犯

**定义**：提供了大量模板，但模板过于理想化，无法在实际工作中填写。

**证据**：
- 4 个模板文件都是纯占位符，没有填写示例
- 模板字段过多（如 full-adr.md 有 10+ 字段），实际使用中工程师会敷衍填写
- 没有提供"最小可行模板"（Minimum Viable Template），即只填 3 个字段就能用的简化版
