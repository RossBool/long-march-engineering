## 二、仍未修复的遗留问题（来自 v1.0.0）

### 🚨 Blocker 1（遗留）：历史隐喻仍然侵入技术决策层

**状态**：❌ **未修复，仅做了表面声明**

**问题描述**：
v1.1.0 在"核心哲学"中声明"历史叙事隐喻仅用于增强记忆和传播性，不替代技术决策"，但**整个 Skill 的命名体系仍然完全依赖历史隐喻**：

- Phase 4 仍然叫"里程碑复盘与路线修正（遵义会议）"
- Phase 6 仍然叫"核心瓶颈限时攻坚（飞夺泸定桥）"
- Phase 7 仍然叫"技术分歧快速裁决（巴西会议、俄界会议）"
- 模板仍然叫"遵义会议模板""泸定桥突击队模板"

**为什么这是 Blocker**：
- **认知绑架**：团队成员在实际会议中会说"我们开个遵义会议吧"，而不是"我们做个架构评审"。隐喻会潜移默化地影响决策——比如"遵义会议"在中国语境中带有"拨乱反正、确立正确领导"的强烈政治暗示，可能导致会议氛围变成"批斗会"而非"技术评审"。
- **文化排他性**：非中国籍团队成员或不熟悉长征历史的成员，无法理解和使用这些术语。RastarGame 作为游戏公司，可能有海外团队或外籍员工。
- **搜索和索引困难**：在文档系统中搜索"架构评审"找不到"遵义会议"，搜索"核心攻坚"找不到"泸定桥"。

**建议修复**：
- 将 9 个 Phase 的**主标题**改为纯技术术语（如 Phase 4: 里程碑复盘与路线修正），历史隐喻移到括号内作为**可选别名**
- 模板名称改为纯技术名称（如"架构评审会议模板"），历史别名仅在"文化层"章节提及
- 或者：彻底删除所有历史隐喻，将 history.md 作为独立文档，不在 SKILL.md 中引用

### 🚨 Blocker 2（遗留）：与已有 Skill 的兼容性声明是空话

**状态**：❌ **未修复，仅增加了声明表格**

**问题描述**：
v1.1.0 增加了一个"与已有 Skill 的兼容性声明"表格，但：

- **没有定义冲突解决机制**：如果 PHP 代码审查 Skill 定义"无上下文架构采纳 = 盲目使用设计模式"，而本 Skill 定义"无上下文架构采纳 = 照搬大厂微服务"，实际使用时以哪个为准？
- **没有定义术语映射表**：两个 Skill 可能用不同术语描述同一概念（如"重写优先综合征" vs " premature refactoring"），没有映射表会导致团队困惑。
- **没有定义加载顺序**：如果用户同时触发 PHP 代码审查 Skill 和本 Skill，哪个优先？

**为什么这是 Blocker**：
- 用户的 SkillHub 中已有多个 Skill，本 Skill 的术语体系如果与其他 Skill 冲突，会导致 AI 在回答时自相矛盾。

**建议修复**：
- 增加"术语映射表"，将本 Skill 的术语映射到其他 Skill 的术语
- 增加"冲突解决规则"（如：本 Skill 仅在迁移场景生效，其他场景以其他 Skill 为准）
- 增加"加载优先级"声明（如：本 Skill 为补充性 Skill，不覆盖其他 Skill 的核心规则）

---

### 🚨 Blocker 3（新增）：文件膨胀导致 AI 上下文溢出

**状态**：❌ **新增问题，v1.0.0 不存在**

**问题描述**：
v1.1.0 文件大小从 4,578 字符暴增到 **23,664 字符**，增长了 **5.17 倍**。

**为什么这是 Blocker**：
- Claude Code 的上下文窗口有限（通常 200k tokens），一个 23k 字符的 Skill 文件在加载时可能占用 6k-8k tokens。
- 如果用户同时加载多个 Skill（如 PHP 代码审查 + Kafka DLQ + 本 Skill），上下文很容易溢出，导致 AI 遗忘重要信息。
- 更致命的是：本 Skill 的 9 个 Phase 中，每个 Phase 都有大量代码示例。当用户只想使用 Phase 4（架构评审）时，AI 被迫加载全部 9 个 Phase 的代码，造成严重的 token 浪费。

**对比**：
- refactor-architecture 采用"渐进式上下文加载"——每 phase 只加载当前 phase 到上下文。
- 本 Skill 虽然学习了"渐进式加载"的概念，但**没有实现**——整个 SKILL.md 是一个单体文件，无法按需加载。

**建议修复**：
- **立即拆分**：将 SKILL.md 拆分为 10 个独立文件（1 个主文件 + 9 个 phase 文件），主文件仅包含索引和通用规则
- 或者：采用"折叠式"结构，每个 phase 的详细内容用折叠标记包裹，AI 按需展开
- 代码示例移至独立文件（`examples/` 目录），Skill 中仅保留链接和简要说明

## 三、新增的严重缺陷（v1.1.0 引入）

### ⚠️ Critical 1（新增）：代码示例未经测试，存在潜在 Bug

**状态**：❌ **新增问题**

**问题描述**：
v1.1.0 增加了 8 个完整代码块，但这些代码**没有经过实际测试**，存在明显的工程问题：

**Bug 1：DataSyncService 没有处理幂等性**
```php
$this->producer->send('order-sync-topic', json_encode([
    'action' => 'create_order',
    'payload' => $args,
]));
```
**问题**：Kafka 消息可能重复消费（至少一次语义），如果新系统没有幂等性检查，同一个订单会被创建多次。

**Bug 2：ShadowTrafficProcess 没有错误处理**
```php
$client = new Client('new-system.internal', 9501);
$client->post('/api/order', $message->payload);
```
**问题**：如果新系统不可用，`$client->post` 会抛出异常，但没有 catch，导致整个消费者进程崩溃。

**Bug 3：VersionRouterMiddleware 没有处理降级**
```php
if ($version === 'v2') {
    $request = $request->withAttribute('route_handler', 'App\\Controller\\V2\\OrderController');
}
```
**问题**：如果 v2 控制器不存在或抛出异常，没有 fallback 到 v1 的机制。

**Bug 4：DataConsistencyCheckCommand 没有处理大表分页**
```php
while (true) {
    $oldOrders = $this->oldDb->table('orders')
        ->where('id', '>', $lastId)
        ->orderBy('id')
        ->limit($batchSize)
        ->get();
```
**问题**：如果 orders 表有 10 亿条记录，这个循环会运行 100 万次，耗时数小时。

**建议修复**：
- **删除所有代码示例**，改为"伪代码 + 关键注意事项"，避免用户直接复制
- 或者：增加醒目的免责声明
- 或者：将代码示例移至独立文件，并标注"TODO: 增加幂等性、错误处理、降级机制"

### ⚠️ Critical 2（新增）：阈值设定主观，未经验证

| 阈值 | 设定值 | 问题 |
|------|--------|------|
| 核心流程中断 > 30 分钟/周 → 重写 | 30 分钟 | 对于高频交易系统已是灾难 |
| 灰度阶段新系统错误率 > 旧系统 × 1.5 | 1.5 倍 | 基数不同效果完全不同 |
| 客户投诉量上升 > 20% → 回滚 | 20% | 投诉受多种因素影响 |
| 休整窗口 3-5 天 | 3-5 天 | 几乎不可能获批 |
| 审计日志缺失 > 0 → 告警 | 0 | 0 容忍会导致告警风暴 |

**建议修复**：每个阈值增加调整指南、校准方法、动态阈值概念。

### ⚠️ Critical 3（新增）："适用性自检"过于理想化

业务方几乎永远不会"书面确认"接受降级。建议改为"业务方知情同意（邮件/会议纪要/IM 记录均可）"。

---

### ⚠️ Critical 4（新增）：ADR 模板过于复杂

提供简化版 ADR（仅包含：问题、决策、原因、影响），将"反对意见记录"改为"风险记录"。

---

### ⚠️ Critical 5（新增）：合规审计部分误导性强

删除 PCI DSS 和 GDPR 的具体检查项，改为"请咨询专业合规团队"的免责声明。
